# 软件著作权申请材料生成器

根据用户提供的软件标题，自动生成完整的软著申请材料。

---

## 生成内容

1. **代码文件**（推荐5-10个），总代码行数4000+
2. **用户手册.md** - 6000字以上，包含至少4个Mermaid图表
3. **信息采集表.md** - 按标准模板格式

---

## 参数定义

| 参数          | 必填 | 说明                                           |
| ------------- | ---- | ---------------------------------------------- |
| `title`       | 是   | 软件全称，如"安全域的系统行为审计日志保护软件" |
| `keyword`     | 是   | 核心关键词，用于生成类名/方法名前缀，如"bgfog" |
| `projectCode` | 否   | 额外的标识前缀，也融入类名/方法名              |
| `targetLines` | 否   | 目标代码行数，默认4000                         |
| `outputDir`   | 否   | 输出目录，默认当前目录                         |

---

## 核心约束

- **零外部依赖**：所有代码仅使用标准库
- **真实业务逻辑**：代码必须真实体现软件标题描述的功能
- **关键词融入**：类名/方法名中必须融入用户指定的关键词
- **合理引用关系**：代码文件之间有合理的import/引用关系
- **编码类型**：优先使用Python + Shell，但也可根据软件类型使用Java、C++、Go等其他语言
- **部署脚本**：优先提供Linux bash部署脚本

## 代码生成规范

### 输出文件命名

| 文件类型 | 命名规则 | 示例 |
|----------|----------|------|
| 用户手册 | `用户手册.md` | 用户手册.md |
| 信息采集表 | `信息采集表.md` | 信息采集表.md |
| 代码文件 | `{keyword}_{module}.{ext}` | `bgfog_models.py`, `bgfog_main.py`, `deploy.sh` |

### 命名规范

- **文件名**：`{keyword}_{module}.{ext}` 如 `bgfog_models.py`, `bgfog_main.py`
- **类名**：`{Keyword}{Function}` 如 `BgfogAuditService`, `BgfogLedger`
- **带项目代码**：`{Keyword}{ProjectCode}{Function}`（可选）
- **方法名**：`{keyword}_{action}` 如 `bgfog_record_event`, `bgfog_verify_integrity`
- **私有方法**：`_{keyword}_{action}`

### 代码组织建议（参考）

根据软件类型和功能，可灵活选择以下模块：

| 模块类型 | 说明 |
|----------|------|
| 数据模型层 | 实体类、数据结构定义 |
| 配置管理 | 配置读取、参数校验 |
| 核心业务逻辑 | 标题描述的核心算法/机制 |
| 策略/算法层 | 权限控制、加密、校验等 |
| 存储层 | 文件/数据存储、轮转、归档 |
| 导出/报表 | CSV/JSON/HTML输出 |
| 服务层 | 业务编排、流程控制 |
| 程序入口 | 初始化、CLI参数处理 |
| 部署脚本 | 环境检查、安装、启动 |

### 模块依赖关系

代码文件之间应有合理的引用关系，例如：

```
main -> config, service, storage
service -> models, core, utils
storage -> models
core -> models
config -> (仅标准库)
models -> (仅标准库)
```

## 用户手册结构（12章）

1. **软件概述**（~300字）- 背景、目标、适用场景
2. **系统架构**（~500字）- Mermaid架构图 + 组件说明
3. **功能说明**（~800字）- 逐项功能详细描述
4. **运行环境**（~300字）- 硬件/软件/网络要求
5. **安装部署**（~600字）- 步骤详解 + Mermaid部署流程图
6. **配置说明**（~500字）- 配置项表格 + 示例
7. **操作指南**（~1000字）- Mermaid操作流程图 + 步骤说明
8. **API接口说明**（~800字）- 核心类/方法参数说明表格
9. **数据流说明**（~500字）- Mermaid时序图
10. **输出结果说明**（~300字）- 输出格式 + 示例
11. **故障排查**（~600字）- 常见问题 + 解决方案表格
12. **版本记录**（~200字）- 变更历史

### Mermaid图表要求（至少4个）

- **架构图**：`graph TD`
- **部署流程图**：`flowchart LR`
- **操作流程图**：`flowchart TB`
- **时序图**：`sequenceDiagram`（数据流）

## 执行流程

1. 分析软件标题，确定软件类型、核心功能和适用编程语言
2. 根据功能复杂度，设计合理的文件数量和模块结构（推荐5-10个）
3. 按依赖顺序生成代码文件（先底层模块，后上层模块）
4. 生成部署脚本（优先Shell脚本）
5. 统计实际代码行数，生成用户手册
6. 生成软著信息采集表
7. 验证所有文件是否满足要求

- [ ] 代码总行数 >= 4000行
- [ ] 代码文件数量合理（5-10个）
- [ ] 每个代码文件 >= 300行（过小则合并，过大则拆分）
- [ ] 类名/方法名包含keyword
- [ ] 用户手册 >= 6000字
- [ ] 手册包含 >= 4个Mermaid图表
- [ ] 代码间引用关系正确
- [ ] 软著信息表按模板格式

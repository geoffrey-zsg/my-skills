# 软件著作权申请材料生成器

根据用户提供的软件标题，自动生成完整的软著申请材料。

---

## 前置信息收集

**开始生成前，必须向用户确认以下信息：**

### 1. 软件全称（必填）
- 完整的软件名称，如"安全域的系统行为审计日志保护软件"

### 2. 申请公司（必选其一）
| 公司 | 简称 | keyword |
|------|------|---------|
| 青雾科技 | 青雾 | `bgfog` |
| 中科麒芯 | 麒芯 | `ickylin` |

**确认方式**：明确询问用户"软著是为哪个公司申请？青雾(bgfog)还是中科麒芯(ickylin)？"

### 3. 项目代号 projectCode（可选）
- 用于代码中类/方法命名的额外标识
- 例如：`audit`, `security`, `manage` 等
- 最终类名格式：`{Company}{ProjectCode}{Function}` 如 `BgfogAuditService`
- 最终方法名格式：`{company}_{projectCode}_{action}` 如 `bgfog_audit_record_event`

## 生成内容

1. **软件/系统设计文档.md** - 设计文档，为代码生成提供参考
2. **代码文件**（推荐5-10个），总代码行数4000+
3. **用户手册.md** - 6000字以上，包含至少4个Mermaid图表
4. **信息采集表.md** - 按标准模板格式

## 参数定义

| 参数 | 必填 | 说明 |
|------|------|------|
| `title` | 是 | 软件全称，如"安全域的系统行为审计日志保护软件" |
| `company` | 是 | 申请公司：`bgfog`(青雾) 或 `ickylin`(中科麒芯) |
| `keyword` | 自动 | 根据company自动确定：`bgfog`或`ickylin` |
| `projectCode` | 否 | 项目代号，用于类/方法命名，如`audit`, `security` |
| `targetLines` | 否 | 目标代码行数，默认4000 |
| `outputDir` | 否 | 输出目录，默认当前目录 |

## 核心约束

- **严格遵循规范**：代码生成必须严格按照本Skill定义的规范执行，不得自由发挥或偏离要求
- **零外部依赖**：所有代码仅使用标准库
- **真实业务逻辑**：代码必须真实体现软件标题描述的功能
- **关键词融入**：类名/方法名中必须严格按命名规范融入关键词
- **合理引用关系**：代码文件之间有合理的import/引用关系
- **编码类型**：优先使用Python + Shell，但也可根据软件类型使用Java、C++、Go等其他语言
- **部署脚本**：优先提供Linux bash部署脚本

## 代码生成规范

### 输出文件命名

| 文件类型 | 命名规则 | 示例 |
|----------|----------|------|
| 设计文档 | `软件设计文档.md` 或 `系统设计文档.md` | 软件设计文档.md |
| 代码文件 | 见下文命名规范 | `bgfog_audit_models.py` |
| 用户手册 | `用户手册.md` | 用户手册.md |
| 信息采集表 | `信息采集表.md` | 信息采集表.md |

### 命名规范

根据是否提供 `projectCode`，采用不同的命名策略：

**无 projectCode 时：**
- **文件名**：`{keyword}_{module}.{ext}` 如 `bgfog_models.py`
- **类名**：`{Keyword}{Function}` 如 `BgfogService`
- **方法名**：`{keyword}_{action}` 如 `bgfog_record_event`

**有 projectCode 时：**
- **文件名**：`{keyword}_{projectCode}_{module}.{ext}` 如 `bgfog_audit_models.py`
- **类名**：`{Keyword}{ProjectCode}{Function}` 如 `BgfogAuditService`
- **方法名**：`{keyword}_{projectCode}_{action}` 如 `bgfog_audit_record_event`

### 代码组织要求（严格执行）

**⚠️ 重要：代码生成必须严格遵循本规范，不得自由发挥。**

根据软件标题分析，从以下模块中选择5-10个组成代码文件：

| 序号 | 模块类型 | 文件名模式 | 说明 |
|------|----------|-----------|------|
| 1 | 数据模型层 | `{keyword}_{projectCode}_models.py` | 实体类、数据结构、枚举定义 |
| 2 | 配置管理 | `{keyword}_{projectCode}_config.py` | 配置读取、参数校验、环境变量 |
| 3 | 核心业务逻辑 | `{keyword}_{projectCode}_core.py` 或 `{keyword}_{projectCode}_service.py` | 标题描述的核心算法/机制 |
| 4 | 策略/算法层 | `{keyword}_{projectCode}_policy.py` 或 `{keyword}_{projectCode}_crypto.py` | 权限、加密、校验逻辑 |
| 5 | 存储层 | `{keyword}_{projectCode}_storage.py` | 文件/数据存储、轮转、归档 |
| 6 | 导出/报表 | `{keyword}_{projectCode}_export.py` | CSV/JSON/HTML输出、报表 |
| 7 | 工具函数 | `{keyword}_{projectCode}_utils.py` | 辅助函数、常量定义 |
| 8 | 程序入口 | `{keyword}_{projectCode}_main.py` | 初始化、CLI参数、Demo |
| 9 | 部署脚本 | `deploy_{projectCode}.sh` | 环境检查、安装、启动 |

**模块选择原则：**
- 必须有：数据模型层、核心业务逻辑层、程序入口
- 推荐有：配置管理、存储层、部署脚本
- 根据功能需要选择：策略层、导出层、工具函数等

**文件数量：** 5-10个代码文件（不含设计文档、用户手册、信息采集表）

### 模块依赖关系

代码文件之间应有合理的引用关系，例如：

```
main -> config, service, storage
service -> models, core, utils
storage -> models
core -> models
config -> (仅标准库)
models -> (仅标准库)
```

## 软件/系统设计文档结构

**在生成代码前，必须先创建设计文档**，内容包括：

1. **软件概述** - 功能定位、目标用户、核心能力
2. **技术架构** - 技术选型、架构模式、模块划分
3. **模块设计** - 各模块职责、接口定义、数据流转
4. **数据模型** - 核心实体、字段定义、关系说明
5. **接口设计** - 类定义、方法签名、参数说明
6. **代码文件清单** - 预计生成的文件名、职责、行数分配
7. **依赖关系图** - 模块间的import/引用关系

**设计文档作用：** 为后续代码生成提供明确的参考依据，确保代码结构合理、功能完整。

**设计文档审批：** 向用户展示设计文档，确认无误后再进入代码生成阶段。

## 用户手册结构（12章）

**⚠️ 重要：用户手册是给最终用户看的操作指南，不是技术文档。减少代码细节，重点描述功能、流程和操作步骤。**

1. **软件概述**（~300字）- 背景、目标、适用场景、能解决什么问题
2. **系统架构**（~500字）- Mermaid架构图 + 各组件功能说明（用户视角，非技术细节）
3. **功能说明**（~800字）- 逐项功能详细描述（能做什么，而非怎么做）
4. **运行环境**（~300字）- 硬件/软件/网络要求
5. **安装部署**（~600字）- 步骤详解 + Mermaid部署流程图
6. **配置说明**（~500字）- 配置项表格 + 示例（告诉用户需要填什么，而非代码实现）
7. **操作指南**（~1000字）- Mermaid操作流程图 + 详细步骤说明（重点章节）
8. **功能使用说明**（~800字）- 各功能模块的使用方法、输入输出、注意事项（不要出现代码/类名/方法名）
9. **数据流说明**（~500字）- Mermaid时序图（数据如何流转，用户视角）
10. **输出结果说明**（~300字）- 输出格式、示例、结果解读
11. **故障排查**（~600字）- 常见问题 + 解决方案表格
12. **版本记录**（~200字）- 变更历史

### 写作原则

- **面向用户**：假设用户不懂编程，不写代码、类名、函数名
- **注重流程**：多描述操作步骤、业务流程
- **图文并茂**：多用Mermaid图表展示流程和架构
- **实用导向**：告诉用户"怎么用这个软件"，而非"软件怎么实现"

### Mermaid图表要求（至少4个）

- **架构图**：`graph TD` - 系统组件及关系
- **部署流程图**：`flowchart LR` - 安装部署步骤
- **操作流程图**：`flowchart TB` - 用户操作步骤（重点）
- **时序图**：`sequenceDiagram` - 数据/业务流转过程

## 软著信息采集表模板

**⚠️ 必须严格按照以下格式生成，不得自由发挥：**

```
软件全称：{title}
软件简称(可无)：{keyword}
版本号：V1.0
软件分类：应用软件
软件完成日期：2025年6月30日
开发的硬件环境：CPU配置、内存配置、硬盘配置
软件运行的硬件环境：CPU配置、内存配置、硬盘配置
开发软件的操作系统：
软件开发环境/工具：
软件运行平台/操作系统：
软件运行支持环境/软件:
编程语言：
开发目的（50字以内）：
面向领域/行业：半导体行业
软件代码行数：{actualLines}
软件主要功能（200字以内）：
软件主要技术特点（100字以内）：

```

**填写说明：**
- 开发/运行硬件环境：根据软件复杂度填写具体配置，如"CPU 4核、内存8GB、硬盘50GB"
- 开发目的：严格控制在50字以内
- 软件主要功能：严格控制在200字以内
- 软件主要技术特点：严格控制在100字以内
- 面向领域/行业：**必须是半导体行业相关**
